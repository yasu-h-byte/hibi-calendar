# 帰国マーカー（✈）が有給（有/P）に変わらない場合の確認手順

最終更新: 2026-04-28

## このドキュメントは何？

「帰国中の実習生・特定技能スタッフに対して、帰国期間を有給に充てる」機能を使って
モーダルから日付を入力したのに、出面入力画面で **✈帰国** のままで **有（P）** に
変わらないときの調査・対処手順です。

エンジニアでなくても、この手順に沿って操作すれば原因が特定できます。

## 過去に発生した事例

### 2026-04-28 リンさんのケース
- 奥寺さんがマニュアルどおりにモーダルから3/9〜14, 16の7日分を有給入力
- 履歴（モーダル内）には正しく記録されていた
- しかし出面入力画面の3月では ✈帰国 のまま変わらなかった
- 原因: 当時の書き込み処理に不具合があり、Firestore上で帰国マーカー（hkフィールド）が
  消えずに残ってしまっていた
- 修正コミット: `cc497f7` （2026-04-28 11:33）

---

## 確認手順

### ステップ1: 管理画面を開く

1. ブラウザ（Chrome推奨）で `https://hibi-calendar.vercel.app/attendance` を開く
2. 管理者としてログインしておく

### ステップ2: 開発者用ツールを開く

ブラウザに「裏側のデータを確認するための画面」を出します。

- **Mac の Chrome**: キーボードで `option + command + I` を同時押し
- **Windows の Chrome**: キーボードで `F12` キー

画面の右側または下側に黒い画面が出てきます。

### ステップ3: 「Console（コンソール）」タブを選ぶ

開発者ツールの上部にいくつかタブが並んでいます。
そのなかの **「Console」** をクリック。

すると、文字を入力できる空欄が出てきます。

### ステップ4: 確認コマンドを貼り付ける

下記のコマンドを **そのままコピー** して、Console の入力欄に貼り付けます。

```javascript
fetch('/api/debug/inspect-att', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'x-admin-password': 'ここに管理者パスワードを入れる',
  },
  body: JSON.stringify({ workerNameLike: 'ヴゥ', ym: '202603' }),
}).then(r => r.json()).then(d => console.log(JSON.stringify(d, null, 2)))
```

### ステップ5: 2つの値を書き換える

貼り付けたコマンドの中で、**2か所だけ書き換えます**。

| 書き換え場所 | 何を入れるか | 例 |
|---|---|---|
| `ここに管理者パスワードを入れる` | 管理者パスワード | （社内で共有しているもの） |
| `'ヴゥ'` | 確認したいスタッフの名前の一部（カナでもベトナム語でも） | `'ファン'`, `'リン'` など |
| `'202603'` | 確認したい年月（YYYYMM形式） | 2026年5月なら `'202605'` |

### ステップ6: Enter キーを押す

書き換えたあと、**Enter** キーを押すと結果が下に表示されます。

### ステップ7: 結果を山田 / 開発担当に共有

結果（黒背景に文字がたくさん出ている部分）を **すべて選択 → コピー** して、
開発担当に貼り付けて送ってください。

---

## 結果の見方（参考）

エンジニアでなくても以下の3点だけは見れば判断できます。

### A. `att_XXX` の中身を見る（XXXはスタッフID）

```
"att_5": {
  "<現場名_5_202603_9>": { "w": 0, "p": 1 },
  "<現場名_5_202603_10>": { "w": 0, "p": 1 },
  ...
}
```

各日付ごとに **`"p": 1`** が入っていれば、有給は記録されています。
**`"hk": 1`** が残っていると、画面では帰国マーカー（✈）として表示されます。

→ **理想の状態**: 該当日に `"p": 1` があり `"hk"` が無い

### B. `pl_XXX` の `designatedLeaves` 履歴を見る

```
"designatedLeaves": [
  { "date": "2026-03-09", "siteId": "...", "kind": "manual-entry", ... },
  ...
]
```

入力した日付すべてが履歴に残っていればOK。
ここに記録があるのに `att_XXX` に `"p": 1` が無い場合、書き込みが
失敗している可能性があります。

### C. siteId（現場ID）が一致しているか

入力時に選択した現場と、`att_XXX` の中身のキーに含まれる現場ID、
`designatedLeaves` の `siteId` が **すべて一致** している必要があります。

別の現場を選択して登録した場合、見ている画面と書き込まれた場所が
ずれている可能性があります。

---

## 対処方法

### パターン1: ハードリロードで解決

ブラウザのキャッシュが古い場合があります。
- **Mac**: `command + shift + R`
- **Windows**: `Ctrl + Shift + R`

これで最新のコードに更新されます。

### パターン2: もう一度モーダルから入力し直す

データに不整合がある場合、同じ日付をもう一度入力すれば
最新の書き込み処理で正しく上書きされます。
（重複カウントにはなりません）

### パターン3: 開発担当に修復を依頼

上記でも直らない場合、Firestore のデータを直接修正する必要があります。
ステップ7で共有した結果を元に、開発担当が修復します。
