## 1. エグゼクティブサマリー

全体評価として、**機能の網羅性は高い**（申請フロー・残数管理・年5日義務・買取・時季指定まで一通り実装済み）一方で、**法令遵守と監査証跡の観点で重大な穴が複数存在**します。特に「誰が職長承認したか追跡不能」「年5日判定基準が実装とドキュメントで矛盾」「画面残数に買取が反映されていない」の3点は、労基署検査または社労士監査で指摘されうるリスクです。

- **Critical（即対応）**: 9件 — 法令違反リスクまたは数字が合わないバグ
- **Major（今週〜今月）**: 17件 — 運用負荷の増加・データ不整合の温床
- **Minor / Info**: 9件 — 中期改善・社労士確認待ち
- **全体健康度**: **C+（要早急な是正、ただし基盤は健在）**

`/api/leave-request` の権限チェックと `/api/leave` の残数計算式の2点を是正すれば、評価は一気に **B+** まで戻せる見込みです。

## 2. ユーザーの3つの関心事への回答

### (1) 申請が正しく反映されているか → **△ 部分的に不安あり**

申請から出面 `p:1` への反映ロジック自体は機能していますが、**「誰が承認したか」「いつ反映されたか」の証跡に穴**があります。

| 観点 | 評価 | 根拠 |
|---|---|---|
| 申請 → 承認の状態遷移 | OK | `leaveRequests.status` で pending/foreman_approved/approved が正しく管理 |
| 出面への `p:1` 書き込み | △ | 承認時のトランザクションが分離しており同時操作で不整合の可能性（finding #22） |
| 取消時の出面クリーン | × | siteId に依存しており、配置変更後の取消で残骸 `p:1` が残るリスク（finding #23） |
| **職長承認者の特定** | **×** | **adminパスワード経由だと workerId=0 が記録され追跡不能（finding #1, #2）** |

特に finding #1（**foreman_approve に権限チェックが一切ない**）は、「他現場の職長が誤って／意図的に承認してしまえる」状態であり、即時対応推奨です。

### (2) 残り日数の管理 → **× 表示が実態とずれている**

ここが最も深刻なエリアです。**画面表示の残数計算式から `buyoutDays`（買取済み日数）が抜け落ちており、買取後も買取分が残数に含まれる**（finding #5, #6）。

```ts
// 実装（app/api/leave/route.ts:1318） — 買取無視
remaining = total - (adjustment + periodUsed)

// 時効処理（同 ファイル :282） — 買取控除
expired = grantDays + carryOver - adjustment - buyoutDays - periodUsed
```

同じレコードに対して「画面表示の残数」と「時効処理の残数」で**別の数字が出る**状態で、これは社労士監査で必ず突かれます。

加えて、

- **スタッフ画面の `plRemaining` が pending 申請を考慮しない**（finding #26）→ スタッフが「あと10日ある」と思って申請したら「残数不足」で却下されるUX
- **多現場重複排除の実装が `lib/leave-compute.ts` と `app/api/leave/route.ts` で重複**（finding #25）→ 将来の乖離リスク

### (3) 事後申請やイレギュラー対応 → **△ 機能はあるが法的整理が未完了**

事後申請（admin/foreman による出面 `p:1` の直接入力）自体は `canAdminEditEntry` で許可されていますが、

- **入力された有給の申請履歴トレースが不能**（finding #10）— `leaveRequests` に対応レコードが作られないため、誰が・いつ・なぜ入力したか追えない
- **帰国中（hk:1）と有給（p:1）の同日設定がガードされない**（finding #9）— 運用解釈次第で違反になりうる
- **補償日（w=0.6）と有給の同日設定もバリデーションなし**（finding #11）
- **中途退職者の未消化分の最終清算ロジックが未実装**（finding #12）— 労基法115条の「退職時の未消化分賃金支払い」義務に対応する処理がない
- **複数日有給の部分取消フローが存在しない**（finding #14）

イレギュラー対応の「機能の入口」は揃っているが、**「出口（記録・清算・通知）」が片肺**という状態です。

## 4. 既存実装で評価できる点

### 健全と判断できた観点

| 観点 | 評価 | 根拠 |
|---|---|---|
| **Firestore 書き込み安全性** | A | 2026-05-07 の事故後に多層防御（lib/firestore-safe.ts, lint, rules, backup）が整備済み。今回の監査で「空マップ書き込み」系の新規 finding はゼロ |
| **付与ロジックの正確性** | A- | 法定付与日数（6/12/24か月…）、勤続換算、繰越上限の処理は仕様通り。findings は付与「タイミング」（退職月除外等）に集中しており、計算式自体は健全 |
| **多現場対応** | B+ | seenDates での重複排除は両箇所で実装済み（実装が分散している点は #25 で指摘あり） |
| **時効処理（2年）の基本ロジック** | B+ | `processExpiry` で `expiredDays` の式は仕様通り。画面残数との不整合（#6）を直せば A 級 |
| **archived レコード除外** | B | 2026-05-08 の修正で繰越計算で正しく除外される。`_archived` のフィールド名統一（#7）が残課題 |
| **PITR 代替バックアップ** | B+ | `backups` コレクションへの日次スナップショット + restore API が稼働中。ID 衝突（#28）以外は健全 |
| **時季指定機能の存在** | B | Phase 5 として designateLeaves 実装済み。法的効力（#19）と UI（#35）が課題だが、機能の基盤はある |
| **買取機能の存在** | B- | Phase 6 として buyoutHistory 実装済み。残数への反映（#5）と法的根拠（#34）が要対応だが、データ構造は健全 |

特筆すべきは、Firestore 書き込み安全性が「事故 → 多層防御 → lint/test 化 → CLAUDE.md 記載」という理想的なサイクルで対処されており、**同種事故が再発しにくい構造**になっている点です。今回の有給機能 findings も、この方法論に従って「コード修正 + lint/test 追加 + CLAUDE.md 記載」のサイクルで一掃することを推奨します。

## 監査結論

機能の幅は十分に揃っており、Firestore 安全性のような土台部分はむしろ模範的です。一方で、**有給領域は「労基法準拠 × 監査証跡」の観点での整理が間に合っていない状態**で、特に「画面残数の不一致」「職長承認の権限チェック欠落」「年5日判定基準のドキュメント矛盾」は、社労士・労基署対応の場面で必ず指摘されるレベルです。

スプリント1〜2（約2週間）の Critical/Major 9件を片付ければ、コンプライアンス上のリスクは大半解消できます。今すぐ着手すべきは **finding #5・#6（残数計算式の統一）** と **finding #1・#2（職長承認の権限チェック）** の2点で、いずれも修正規模は小〜中、影響範囲は限定的です。

ご指示いただければ、スプリント1の4件から順に着手します。
