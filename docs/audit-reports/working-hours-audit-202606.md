## 2. 根本原因

問題は「1つのバグ」ではなく、**同じ労働時間の計算が複数の関数に重複実装され、それぞれ微妙に異なる前提を持っている**ことです。4つの原因が連鎖しています。

### 2-1. 残業判定ロジックの二重実装（最重要・労基法32条の2）

変形労働時間制の3層残業判定（日 / 週 / 月）が、**2つの関数で別々に実装**されています。

| 関数 | 用途 | 週残業の判定 |
|------|------|------------|
| `calculateOvertimeSummary` (compute.ts:1463) | 社労士提出用 Excel | `weekPrescribed` を集計し、**週所定40h以下→40h超を残業 / 週所定40h超→その所定超を残業**（変形労働に正しく対応） |
| `calculateVietnameseSalary` (compute.ts ~1742) | **実際の給与計算** | **固定40hハードコード** |

給与計算側（compute.ts:1895）の該当コード:

```ts
// calculateVietnameseSalary — 週残業判定
totalWeeklyOT += Math.max(0, weekRegular - 40 - weekDailyOT)   // ← 常に 40h 固定
```

社労士提出 Excel 側（compute.ts:1596-1602）の正しいコード:

```ts
// calculateOvertimeSummary — 週残業判定
if (weekPrescribed <= 40) {
  weeklyStatutoryOT = Math.max(0, weekActual - 40 - weekDailyOT)
} else {
  weeklyStatutoryOT = Math.max(0, weekActual - weekPrescribed - weekDailyOT)  // 変形対応
}
```

**結果:** 笹塚現場（週6日=週42h所定）に配置されたスタッフ（ID 203/204/205 等）の給与計算では、週42h所定なのに**40h超の2h/週が「幻の残業」として割増支給**される。労基法32条の2では、事前に定めた各週の所定（42h）を超えた分のみが残業のはずです。社労士提出 Excel は正しく42hで判定するため、**給与額と提出書類の残業時間が食い違います。**

### 2-2. 実労働時間の計算が帳票ごとに違う（労基法34条 休憩）

「1日の実労働時間 = 始業〜終業 − 休憩」の **休憩の引き方**が、給与計算と実労働明細 Excel で異なります。

| 帳票 | 休憩控除のロジック | コード |
|------|------------------|--------|
| 給与計算 | `calcActualHours` — **休憩フラグ b1/b2/b3 が立っている分だけ控除** | compute.ts:1821-1823 |
| 実労働明細 Excel | `computePrescribedFromSchedule` — **現場の全休憩を固定で控除**（フラグ無視） | export.ts:1805-1816 |

`calcActualHours` の核心部分（types/index.ts:164-166）:

```ts
if (entry.b1) totalMinutes -= morningMin    // b1 が真のときだけ朝休憩を引く
if (entry.b2) totalMinutes -= lunchMin      // b2 が真のときだけ昼休憩を引く
if (entry.b3) totalMinutes -= afternoonMin  // b3 が真のときだけ午後休憩を引く
```

**グエン201（5月IHI現場・07:30–16:30, b1=0/b2=1/b3=0）の具体例:**

- 実労働明細 Excel: 540分 − 全休憩120分（朝30+昼60+午後30）= **7.0h**
- 給与計算: 540分 − 昼60分（b2のみ立っている）= **8.0h**

**同じ日で 1.0h ズレます。** 社労士が「実労働明細は7h、給与は8h、どっちが正しいのか」と照合できないのは、このためです。

### 2-3. 休憩フラグ未設定時の過大計上リスク（実害が潜在）

§2-2 のロジックには、より危険な副作用があります。`calcActualHours` は **「休憩フラグが立っていなければ休憩を引かない」**＝implicit-default-no-break 設計です。フラグが `0` や `undefined`（未入力）だと、**休憩が一切控除されず実労働が膨らみます。**

- グエン201はスマホ入力で休憩を明示的に入力していたため、たまたま正しく8hが出ている。
- しかし**職長/管理者が PC で「仕事」エントリを手入力した場合**、`app/api/attendance/foreman/route.ts:203` の作成処理は `st/et/b1/b2/b3` を保存しないため、フラグが全て undefined になり得ます。
- その場合、07:30–16:30 が **540分−0=9.0h** と計上され、所定7hに対し **+2h/日が残業扱い**になる危険があります。

**これは過大支給（会社の過払い）に直結します。** 実態は Firestore の実査が必要ですが（§6）、5月分の `att_202605` に「st/et あり・b1/b2/b3 すべて未設定」のエントリが何件あるかで影響額が決まります。

### 2-4. 残業フィールド `o` と st/et の二重定義

旧ルール（〜202604）では `o` フィールドが「日次残業時間」を表していました。新ルール（202605〜）では st/et から残業を自動導出するため `o` は原則不要ですが、**入力 UI 側で両方の同時入力を禁止する仕組みがなく**、混在しうる状態です。

- **給与計算**: st/et があれば `o` を無視（実害なし、ただし暗黙的）
- **実労働明細 Excel の残業列**: 常に `e.o` を表示（export.ts:1829）

→ 給与計算は「st/et 由来の所定外」、実労働明細は「入力欄の o」を出すため、**同じ「残業」という列名で違う数字**が並びます。これも③の「出どころ不明」の一因です。

### 2-5. 勤務予定シフトの現場判定バグ

`inferPrimarySiteId`（export.ts:1622）が月次配置替え（massign）を引くキーを間違えています:

```ts
const mOver = massign[ym]   // ← massign の実キーは "siteId_ym" 形式（例 "sasazuka_202605"）
```

実際の massign は `ihi_202605` のような複合キーのため `massign[ym]` は常に undefined となり、月次配置替えが効かずデフォルト配置にフォールバックします。グエン201は偶然デフォルト先頭が正しい現場のため正常に見えますが、**笹塚へ5月だけ配置替えされた 203/204/205 等は、勤務予定シフト Excl で誤った現場・誤った所定時間が出る**可能性があります。

## 5. 社労士への回答案

> **① 勤務予定シフトの「7h×週6日=42h」について**
>
> 当社は1ヶ月単位の変形労働時間制を採用しており、笹塚現場のように月〜土6日稼働の週は、協定上その週の所定を42時間と定めています。シフト表に表示される42hは「その週の所定労働時間」であり、超過ではありません。表に「週所定」の欄がなく誤読を招いたため、勤務予定シフト表に週所定列を追加します。

> **② 給与計算の「7h×20日=140h」について**
>
> 給与計算は変形労働の月法定上限（暦日数×40÷7時間）を基準に、日・週・月の3段階で法定外残業を判定しています。140hはこの枠内です。①の週42hと②の月140hは別レイヤーの数値で、現状の帳票では相互参照ができないため、実労働明細に両者を突き合わせられる欄を追加します。

> **③ 実労働明細の残業列の出どころについて**
>
> ご指摘の通りでした。確認の結果、(a) 実労働明細と給与計算で休憩の控除方法が異なり（明細=全休憩固定控除、給与=実際に取得した休憩のみ控除）、同じ日で実労働が最大1時間ずれていました。また (b) 明細の残業列が「出面入力の残業欄」を、給与計算が「3層判定の法定外残業」を表示しており、定義が不一致でした。両者を給与計算と同一ロジックに統一し、明細上で「実労働h／所定h／法定外残業（日・週・月）」を一貫表示するよう是正します。
>
> なお、給与計算ロジック側に週残業の判定を一律40hで行っていた箇所があり、笹塚等の週42h所定現場で過大に残業計上される可能性がありました。これを変形労働対応へ修正します。グエンさん本人の5月支給額に過誤は確認されていませんが、対象となりうるスタッフの5月分を再計算し、差額が生じた場合は是正します。
