# ロール・認証・権限

## 認証方式
- 個人パスワード: 役員・事務（政仁、奥寺、チイ）は個人パスワードで直接ログイン（名前選択不要）
- 職長共有パスワード: 環境変数 ADMIN_PASSWORD。ログイン後に名前を選択（職長のみ表示）
- スーパー管理者: 環境変数 SUPER_ADMIN_PASSWORD（直接ログイン、名前選択不要）
- スタッフ: トークンURL（ログイン不要）

## ロール一覧
| ロール | 日本語表示 | 判定方法 | 対象者 |
|--------|-----------|----------|--------|
| admin | 管理者 | デフォルト / スーパー管理者 | 靖仁、その他 |
| approver | 事業責任者 | workerId === 1 | 政仁 |
| foreman | 職長 | 現場のforeman設定 | 大川、白戸、日比大介 |
| jimu | 事務 | jobType === 'jimu' | 奥寺 |

## ロール変更方法
- 人員マスタの「職種」を変更すると連動（役員→admin、職長→foreman、事務→jimu）
- 政仁さん（ID:1）はハードコードでapprover固定

## 承認権限のロール制御（有給申請・帰国申請）

「現場レベルの作業は職長、最終承認は事業責任者」という原則をボタン表示レベルで実装。

### 承認ボタンの表示権限
| ボタン | admin | approver | foreman | 備考 |
|--------|-------|----------|---------|------|
| 職長承認 | ✓ | ✓ | ✓ | pending 状態の申請に対して表示 |
| 最終承認 | ✓ | ✓ | | foreman には「最終承認待ち」のラベルのみ表示 |
| 却下 | ✓ | ✓ | ✓ | 全員可能 |

- 適用画面: ダッシュボードの勤怠申請カード、および `/leave` の申請タブ
- foreman は foreman_approved 状態の申請に対して操作不可（読み取り専用で「最終承認待ち」と表示）
- 職長が直接「最終承認」する経路は UI レベルで遮断される

## 通知（/api/notifications）

`leaveRequests`（有給申請）と `homeLongLeave`（帰国申請）の両方を、`pending` + `foreman_approved` の合計件数でカウントする。

### 通知メッセージ書式
- `有給申請 X件 （職長待ちN / 最終承認待ちM）`
- `帰国申請 X件 （職長待ちN / 最終承認待ちM）`

### ロール別の意味づけ
- admin / approver: 職長待ち・最終承認待ちの両方が自分の対応対象（最終承認は admin / approver のみが押せるため）
- foreman: 自分の対応待ちは pending（職長待ち）分のみ。foreman_approved 分は参考表示
