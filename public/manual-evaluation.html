第1部　評価制度の全体像

1.1 目的と背景 全員

本評価制度は、外国人スタッフ（技能実習生・特定技能）の処遇を客観的かつ公平に決定するための仕組みです。 複数の職長と事業責任者・管理者の評価を組み合わせ、各評価者の実際の接点（共働日数）に応じた重み付けで、特定の人の主観に偏らない結果を出すことを目的としています。

評価結果が反映されるもの
  • 時給の昇給額（評価ランクと勤続年数で自動算出）
  • 育成方針の判断材料（どの分野を伸ばすか）
  • 更新契約・雇用継続の判断材料

1.2 評価対象・評価者・サイクル 全員

評価対象

在留資格を持つ外国人スタッフ全員（日本人スタッフは対象外）。 原則として入社記念日の前後1ヶ月で年1回評価を実施します。

評価者の構成（5名）

区分担当者役割
管理者靖仁さん（workerId=0）セッション作成・承認・生活態度の評価
事業責任者政仁さん（workerId=1）最終承認・業務評価3カテゴリ
職長 3名白戸さん・大川さん・大介さん業務評価3カテゴリ
なぜ生活態度だけ靖仁さんが評価？
寮の運営・近隣からの苦情対応・寮ルール管理は管理部門の担当領域です。職長は現場での業務しか把握しないため、現場接点のない生活面を評価できる立場にありません。スコープを分けて各自が責任を持って評価できる範囲だけを採点する設計にしています。

評価サイクル

  • 年1回・入社記念日基準
  • セッション作成から承認まで通常2〜3週間
  • 承認結果は翌月の給与（昇給）に反映

1.3 評価フローの全体像 全員

ステップ担当やること該当タブ
①管理者評価セッション作成（対象スタッフと評価者を選択）一覧
②〜⑥職長3名 + 政仁さん + 靖仁さん各自ABC評価を入力して提出評価入力
↓ 全員提出が完了すると、自動で「最終確認待ち」に遷移 ↓

⑧システム承認時に手動スコア・合計スコア・ランク・推奨昇給額を自動算出して保存—

2.2 ABC判定の目安（カテゴリ別） 全員

迷ったらBが基本。明確に上回る／下回る具体例があるときだけAまたはCを付けてください。

🗣 日本語能力（3項目 × ×1.0 = 9点満点）

項目ABC
指示理解一度で正確に理解し、確認の質問もできる概ね理解するが、たまに聞き返しが必要繰り返し説明が必要。複雑な指示は通じにくい
報告・連絡自分から日本語で報告・連絡ができる。問題発生時も的確に伝えられる聞かれれば答えられるが、自発的な報告は少ない通訳が必要な場面が多い。報告が遅れることがある
安全用語危険予知・安全指示を理解し、他のスタッフにも伝達できる基本的な安全用語（危ない・止まれ・ヘルメット等）は理解している安全に関する指示の理解が不十分。安全面でのリスクがある

💼 勤務態度（4項目 × ×1.5 = 18点満点 — 最重要）

項目ABC
時間厳守・出勤無遅刻・無欠勤（有給除く）。始業前に準備を済ませている年1〜2回の遅刻程度。基本的には時間を守る遅刻・無断欠勤が目立つ。時間管理に課題がある
安全意識ヘルメット・安全帯の着用を自ら徹底。周囲の安全にも気を配る注意すれば守る。基本的なルールは理解している繰り返し注意が必要。安全装備の着用忘れや不安全行動がある
協調性・チームワーク他のスタッフと積極的に協力し、後輩の面倒も見る。現場の雰囲気を良くする与えられた仕事を問題なくこなす。チームワークに支障はない単独行動が多い。コミュニケーションが不足し、チームワークに課題がある
指示遵守職長や先輩社員の指示を正確に守り、自己判断で勝手に変えない。不明点は必ず確認する基本的に指示は守るが、たまに自己判断で省略・変更することがある指示を守らないことが目立つ。自分のやり方で勝手に進めてしまうことがある

🔨 職業能力（3項目 × ×1.0 = 9点満点）

項目ABC
技能レベル一人で任せられる作業が多い。難易度の高い作業もこなせる指示を受ければ一通りの作業ができる。標準的なレベル常に監督が必要。基本的な作業でもミスが多い
作業速度・品質日本人と同等以上のスピードと品質。手直しがほとんど不要概ね問題ないレベル。時々確認や手直しが必要スピード・品質に改善が必要。手直しが頻繁に発生する
段取り・準備次の工程を予測して自ら材料・道具を準備できる。段取りが良い言われれば段取りできる。基本的な準備はできる段取りの意識が低い。指示待ちになることが多い

🏠 生活態度（3項目 × ×1.0 = 9点満点）

項目ABC
近隣配慮近隣からの苦情ゼロ。騒音・駐車等に自ら配慮し、他のスタッフにも注意できる基本的なルールは守る。年1〜2回注意を受ける程度近隣からの苦情がある。騒音・マナー違反を繰り返し注意される
ルール遵守ゴミ出し・分別・当番など寮のルールを率先して守る。他のスタッフにも声かけできる言われれば守る。基本的なルールは理解しているルールを守らないことが多い。繰り返し注意が必要
住環境維持部屋・共用部をいつもきれいに保つ。整理整頓が行き届いている普通の清潔さは保っている。時々散らかっていることがある部屋や共用部が不衛生。清掃を怠ることが多い

第3部　評価者ウェイトの仕組み

3.1 なぜウェイトが必要か 管理者事業責任者

全員の意見を単純多数決で扱うと、実際に対象スタッフと一緒に働いていない職長の意見が、毎日一緒の職長と同じ重みで反映されてしまいます。 これでは公平とは言えません。 そこで「対象スタッフと過去1年でどれだけ一緒に働いたか」を元に、各評価者の意見をどの程度反映するかを自動算出するウェイト機能を導入しています。

3.2 役割別の基準ウェイト 管理者事業責任者

評価者固定値説明
靖仁さん（admin）0.5（固定）現場接点が少ないため抑制した値。担当する生活態度の評価には十分な接点があるため、その評価のみで使用される
政仁さん（approver）0.7（固定）事業責任者として全般を見る立場。現場には日常的に出ているが個別スタッフとの密着度は職長より低い
職長3名0.1 〜 1.0（自動）対象スタッフとの直近1年の共働日数で自動算出（次節）

3.3 職長のウェイト計算式 管理者事業責任者

計算式

monthsWithData = 過去365日 のうち att データのある月数（0〜12）
dynamicCap     = max(20, round(200 × monthsWithData / 12))
yearDays       = 評価日から過去365日の共働日数

IF yearDays == 0:
  weight = 0.1   ← 「この人を知らない」職長の特例
ELSE:
  weight = 0.3 + 0.7 × (min(yearDays, dynamicCap) / dynamicCap)
範囲: 0.1（ゼロ日特例）または 0.3〜1.0（共働あり）

なぜ動的キャップ？

旧システム稼働前（2025年9月以前）の出面データはほぼ存在しません。 12ヶ月分のフル日数を固定キャップにすると、データの少ない期間しか評価対象にできないスタッフの主担当職長まで不当に低いウェイトになってしまいます。 そこで「データのある月数」に按分してキャップを縮小し、主担当職長が8ヶ月分のデータでも 100% に到達できるよう調整しています。

具体例（評価日: 2026-05-07）

例1: 主担当職長 — 大川さん × グエン・ミン・サンさん

monthsWithData8ヶ月（2025-10〜2026-05に出面データあり）
dynamicCapmax(20, round(200×8/12)) = 133日
yearDays74日
weight0.3 + 0.7 × (74/133) = 0.69

例2: 接点ゼロ — 白戸さん × HFUスタッフ

yearDays0日（別現場で一度も共働なし）
weight0.1（ゼロ日特例）
共働日数の定義
  • 対象スタッフが「その評価者が月の職長として責任を持っていた現場」で実出勤した日数
  • 月の職長は mforeman[siteId_ym]（月別オーバーライド）→ site.foreman（デフォルト）の順で解決
  • 有給・休み・帰国中・試験などは共働にカウントしない（実出勤のみ）

3.4 ウェイトの確認・再計算 管理者事業責任者

確認方法

  • 承認画面の比較表ヘッダ — 各評価者の名前の下に w=0.70 等で表示
  • ホバーで「過去365日 共働X日／うち直近90日 Y日」が補足表示
  • A4印刷ページにも反映される

再計算ボタン

出勤データを後から修正した場合などに、進行中セッションのウェイトを最新の出勤実績で再計算できます。承認画面の「🔄 ウェイト一括再計算」または個別セッションの「🔄 ウェイト再計算」ボタンを使用。承認済みセッションのウェイトは原則変更しません（過去の評価結果を変えないため）。

4.4 提出状況・リマインダー 管理者事業責任者

進行中（収集中・最終確認待ち）のセッションだけを2カラムで表示。提出率や経過日数が一目で分かります。

自動通知

7日以上提出のないセッションがあると、ダッシュボードに通知バナーが表示されます（管理者向け）。

4.5 承認フロー 管理者事業責任者

1
「承認」タブ → 最終確認待ちのセッション一覧
2
「確認・承認」ボタン → 詳細画面に遷移し、加重平均で最終ABCがプリフィルされる
3

4
最終コメントを記入（任意・スタッフへのフィードバック）
5
「承認する」→ ランク・推奨昇給額が自動算出されて保存

加重平均プリフィルの計算

各項目について、スコープ内評価者の A=3 / B=2 / C=1 を重みで加重平均し、しきい値でABCに戻します。 

weighted_avg = Σ (grade × weight) / Σ weight  （スコープ内評価者のみ）
最終ABC:
  ≥ 2.5  → A
  ≥ 1.5  → B
  < 1.5  → C
計算例：政仁(0.7)=A・白戸(0.1)=B・大川(1.0)=A・大介(0.1)=B
weighted_avg = (3×0.7 + 2×0.1 + 3×1.0 + 2×0.1) / (0.7+0.1+1.0+0.1) = 5.5 / 1.9 = 2.89 → A
承認権限：承認は 管理者（靖仁さん）または事業責任者（政仁さん） のみ可能。職長は承認できない。

4.6 詳細モーダル・A4印刷 全員

詳細モーダル

一覧タブ・履歴タブから「詳細」ボタンで開く読み取り専用ビュー。

  • スタッフ情報（氏名・在留資格・入社日・勤続年数）
  • 提出状況バッジ
  • 評価者ウェイト一覧
  • 出勤指標（出勤率・残業平均・有給取得・皆勤ボーナス）
  • 比較表（各評価者の評価 + 最終評価）
  • 承認済みの場合: 合計スコア・ランク・推奨昇給額
  • 各評価者のコメント + 最終コメント

A4印刷（承認済みのみ）

詳細モーダル右上の 「🖨 A4印刷」 ボタンで、A4縦1枚に整形された個人評価表が新タブで開きます。ブラウザの印刷ダイアログで紙印刷 or PDF保存が可能。

印刷出力内容
  • ロゴ + タイトル + 評価日・承認日
  • スタッフ情報・出勤指標
  • 13項目 × 5評価者 + 最終評価のグリッド
  • 合計スコア・ランク・推奨昇給額
  • 全評価者のコメント全文 + 最終コメント

4.7 履歴・再計算 管理者事業責任者

履歴タブ

承認済み評価をすべて表示。所属会社・勤続年数・在留資格・時給・ランク・推奨昇給などでソート可能。各行から詳細モーダルとA4印刷にアクセスできます。

再計算 API（自己修復ツール）

評価ロジックを将来変更した場合などに、過去の承認済み評価を最新ロジックで再計算するためのバックエンドAPIです。

使用例（管理者がブラウザDevToolsで実行 or curl）： 
# 個別セッション再計算
POST /api/evaluation
{ "action": "recalculateApproved", "evaluationId": "204_2026-05-07" }

# 全承認済みを一括再計算
POST /api/evaluation
{ "action": "recalculateApproved", "all": true }
finalScores（評価内容）は変更されず、manualScore / totalScore / rank / raiseAmount のみ最新ロジックで再計算されます。再計算した評価は recalculatedAt フィールドに日時が記録されます。

第5部　スコアと昇給

5.1 手動スコアの計算式 全員

4カテゴリそれぞれで A=3 / B=2 / C=1 で素点を集計し、重み係数を掛けて合算します。 

japaneseSum  = 3項目の素点合計（最大9）
attitudeSum  = 4項目の素点合計（最大12）
skillSum     = 3項目の素点合計（最大9）
livingSum    = 3項目の素点合計（最大9）

manualScore  = japaneseSum × 1.0
             + attitudeSum × 1.5
             + skillSum    × 1.0
             + livingSum   × 1.0

最大値: 9×1.0 + 12×1.5 + 9×1.0 + 9×1.0 = 45.0
計算例：日本語=Bx3 / 勤務=Ax3+B / 職業=B+B+A / 生活=Bx3 の場合
手動スコア = 6×1.0 + (9+2)×1.5 + 7×1.0 + 6×1.0 = 6 + 16.5 + 7 + 6 = 35.5

5.2 出勤指標と皆勤ボーナス 全員

スコアには評価日の過去12ヶ月の出勤実績から算出した皆勤ボーナス（最大+3点）が加算されます。

出勤率皆勤ボーナス
100%（無欠勤）+3
95%以上+2
90%以上+1
90%未満0
出勤率の計算 
出勤率 = (実出勤日 + 有給日 + 試験日) / 期待出勤日 × 100  (上限100%)
  • 期待出勤日: 月の所定日数の合計（カレンダーで算出）から、雇用前・退職後・一時帰国期間・14日以上の連続無出勤期間 を除外
  • 分子: 実出勤＋有給＋試験（補償日 w=0.6 は分子・分母どちらにも含めない）

5.3 ランク判定（S/A/B/C/D） 全員

合計スコア = 手動スコア + 皆勤ボーナス（最大48点）からランクを決定します。

ランクしきい値満点比意味
S≥ 3981%+非常に優秀（特別な活躍）
A≥ 3267%+優秀（期待を上回る）
B≥ 2552%+標準（期待通り）
C≥ 1735%+要改善
D< 1735%-不適格（改善の見込みなし）

第6部　運用ルール

6.1 バイアス防止 全員

  • 提出前は他人の評価を見ない：自分が提出するまで他評価者の結果は画面に表示されない
  • 事前打ち合わせ禁止：「みんなBにしよう」などの根回しは制度の意義を損なう
  • 具体的な事実ベースで採点：印象ではなく「いつ・何があったか」で判断
  • カテゴリスコープを守る：自分が担当しないカテゴリは画面に出ないので、誤評価のリスクなし

6.2 評価期間中の禁止事項 全員

  • 評価対象スタッフへの「自分はこう評価する」の事前告知
  • 他評価者の評価内容を本人に漏らす
  • 承認前のスコアやランクを本人に伝える
  • システム外で評価結果を集計し直す（公式な記録は本システムのみ）

6.3 守秘義務（給与情報の取扱い） 管理者事業責任者

推奨昇給額・実際の時給情報はスタッフ本人と管理職以外には開示しません。 評価入力ページや承認画面では時給を表示しない設計になっています。 昇給履歴タブも管理者と事業責任者のみアクセス可能で、職長は閲覧できません。

第7部　トラブル対処・FAQ

7.1 よくある質問 全員

Q. 評価を間違えて提出してしまった

承認前なら「修正」ボタンから何度でも更新できます。承認後はロックされるため、管理者に相談してください。

Q. 担当外のカテゴリが表示されない

仕様です。靖仁さんは生活態度のみ、他の評価者は業務系3カテゴリ（日本語・勤務態度・職業能力）のみ表示されます。

Q. 私のウェイトが他の職長より低いのはなぜ？

対象スタッフとの過去365日の共働日数で自動算出されます。実際に一緒に働いていない期間が多いと低くなる仕組みで、評価結果の公平性を保つためです。

Q. 接点がないスタッフを評価しなければならない

コメント欄に「接点がないため評価困難」と書いて、全てBで提出してOKです。ウェイトが0.1（特例）になるため、評価結果への影響は最小限になります。

Q. 加重平均でBになった項目を、最終評価でAに変えていい？

はい。プリフィルはあくまで初期値の提案です。承認者の判断で最終的なABCを決定できます。ただし大幅に変える場合は最終コメントに理由を残してください。

Q. 既に承認した評価のスコアやランクが間違っている

管理者に連絡してください。「再計算API」（4.7参照）で最新ロジックで再計算するか、極端な不整合の場合はFirestoreの直接修正が必要になることがあります。

7.2 不整合発生時の修復 管理者

ロジック変更後に過去データを更新したい

評価ロジック（重み・ランクしきい値・昇給テーブルなど）を変更した場合、過去の承認済み評価は古いロジックで計算された値のまま保存されています。最新ロジックで再計算するには： 

# 全承認済みを一括再計算
POST /api/evaluation
Headers: x-admin-password: <管理者パスワード>
Body: { "action": "recalculateApproved", "all": true }
注意：再計算で rank や raiseAmount が変わる可能性があります。給与計算に影響する可能性があるため、本番反映前に該当スタッフに事前説明することを推奨します。

評価ロジックの修正箇所

評価制度の重み係数・ランクしきい値・昇給テーブルは、すべて lib/evaluation-config.ts に集約されています。フロント・バックエンド両方がこのファイルから import するため、片方だけ更新するドリフト事故は構造的に発生しません。

評価ロジックの単一の真理ソース
lib/evaluation-config.ts をのみ編集すること。
EVALUATION_WEIGHTS、RAISE_TABLE、calculateManualScore、calculateRank、getRaiseAmount がここで定義されています。
